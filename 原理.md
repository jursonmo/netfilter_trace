# nftracepath 实现原理

这个项目的目标不是“静态分析一份防火墙规则，然后猜某条流会怎么走”，而是在目标 Linux 机器上针对用户指定的 IPv4 TCP/UDP 流临时插入探针规则，让真实数据包经过 netfilter 时由内核吐出 trace/log 事件，再把这些事件解析成可读的路径结果。

简单说：

1. 用户输入一个流：协议、源 IP、目的 IP、可选源端口、可选目的端口、可选入接口。
2. 程序选择 `nft` 或 `iptables` 后端。
3. 程序临时安装精确匹配这条流的 TRACE/LOG 规则。
4. 数据包真实经过内核 netfilter hook。
5. 内核输出 trace/log。
6. 程序监听这些输出，过滤出本次 run id 和这条流对应的事件。
7. 程序清理临时规则，输出命中的表、链、规则、接口和最终结果。

## netfilter 的底层路径

Linux 防火墙的底层核心是 netfilter。无论上层用 `nftables` 还是 `iptables` 管规则，数据包都会经过内核中的几个关键 hook：

```text
入方向包:

网卡收到包
  -> PREROUTING
  -> 路由判断
       -> INPUT        -> 本机进程
       -> FORWARD      -> 转发
  -> POSTROUTING       -> 发出网卡

本机发出的包:

本机进程
  -> OUTPUT
  -> 路由判断
  -> POSTROUTING
  -> 发出网卡
```

这个项目就是在这些 hook 上插临时规则：

- 在 `PREROUTING` 或 `OUTPUT` 的早期位置打开 trace。
- 在 `INPUT`、`FORWARD`、`POSTROUTING` 的靠后位置插 probe，判断包最后是进本机、被转发，还是发出了哪块网卡。
- 如果中途遇到 `drop` 或 `reject`，就把结果标成丢弃或拒绝。

## 为什么不能只靠静态分析规则

理论上可以读取 `nft list ruleset` 或 `iptables-save`，然后模拟一遍规则匹配。但实际很难可靠：

- 规则可能包含 set、map、mark、conntrack state、接口、路由结果等运行时状态。
- NAT 会改变地址和端口，后续规则看到的五元组可能和原始输入不同。
- chain jump/goto/return、默认 policy、不同表的 hook priority 都会影响执行路径。
- 内核模块扩展匹配条件很多，用户态复刻一遍成本很高。

所以本项目采用运行时追踪：让真实内核告诉我们这个包实际走过了哪些规则。

## nftables 后端如何知道命中了哪些规则

nftables 有比较适合这个项目的原生 trace 能力。核心机制是：

```nft
meta nftrace set 1
```

当一个包命中这条规则后，内核会给这个包打上 `nftrace` 标记。之后这个包继续经过 nftables 规则集时，内核会通过 `nft monitor trace` 输出 trace 事件。

项目里的 nft 后端做了这些事：

1. 创建临时表：

```text
inet nftracepath_<runid>
```

2. 创建临时 chain：

```text
trace_prerouting   hook prerouting  priority -300
trace_output       hook output      priority -300
final_input        hook input       priority 300
final_forward      hook forward     priority 300
final_postrouting  hook postrouting priority 300
```

3. 在早期 chain 里安装精确匹配规则：

```nft
ip saddr <src> ip daddr <dst> meta l4proto <tcp|udp> ...
limit rate <rate> burst <burst> packets
meta nftrace set 1
counter
comment "nftracepath:<runid>:trace-prerouting"
```

如果用户没有指定 `--in-iface`，还会在 `OUTPUT` hook 安装类似规则，用于追踪本机发出的包。如果指定了 `--in-iface`，就不装 `OUTPUT` trace，因为本机发出的包没有“入接口”，否则可能把同五元组的本机出向包也混进来。

4. 程序启动：

```sh
nft monitor trace
```

5. 内核输出类似事件：

```text
trace id 1234 inet filter input packet: iif "eth0" ip saddr 192.0.2.10 ip daddr 198.51.100.20 tcp sport 12345 tcp dport 443
trace id 1234 inet filter input rule tcp dport 443 accept # handle 7 (verdict accept)
trace id 1234 inet nftracepath_abcd final_input packet: iif "eth0" ...
```

这里最重要的是 `trace id 1234`。同一个包在 nft trace 输出中的多行事件会共享同一个 trace id。项目用它把 packet/rule/verdict/policy 事件串成一条路径。

### nft 规则文本如何还原

`nft monitor trace` 的 rule 事件里通常会带 `# handle <number>`。项目安装完临时规则后会执行：

```sh
nft --handle list ruleset
```

然后解析出：

```text
family/table/chain/handle -> rule text
```

当 trace 行里出现同一个 handle 时，就能反查完整规则文本。相关逻辑在 `internal/trace/parse.go`：

- `parseNFTRuleset` 建立 handle 到规则文本的映射。
- `parseNFTTraceLine` 解析 trace 行，提取 trace id、family、table、chain、handle、verdict、接口。

### nft 如何判断最终去向

项目额外创建了三个 final chain：

- `final_input`：包到了这里，说明它进入本机 `INPUT` 路径。
- `final_forward`：包到了这里，说明它进入转发路径。
- `final_postrouting`：包到了这里，说明它准备从某个出接口发出。

这些 chain 的规则不改变包，只是 `counter comment`。因为包之前已经被 `meta nftrace set 1` 标记，进入这些 final chain 时也会被 `nft monitor trace` 看到。程序看到 `final_input`、`final_forward`、`final_postrouting` 这些 chain 名，就能推断结果：

- `final_input` -> `local`
- `final_postrouting` -> `egress`
- 规则 verdict 是 `drop` -> `drop`
- 规则 verdict 是 `reject` -> `reject`
- 有事件但没看到终态 -> `unknown`
- 完全没看到事件直到超时 -> `timeout`

## iptables 后端如何实现

iptables 的可观测能力比 nftables 粗糙。它主要依赖两个内核 target：

- `TRACE`：让内核在后续规则遍历中向 kernel log 打印 trace 信息。
- `LOG`：在指定位置主动打印一条带前缀的日志，用来当 final probe。

项目中的 iptables 后端主要面向 `iptables-legacy`，流程如下。

### 1. 在 raw 表打开 TRACE

项目会插入：

```sh
iptables -t raw -I PREROUTING 1 ... -j TRACE
```

如果没有指定 `--in-iface`，还会插入：

```sh
iptables -t raw -I OUTPUT 1 ... -j TRACE
```

为什么是 raw 表？因为 raw 表的 `PREROUTING` 和 `OUTPUT` 位置很早，可以尽量早地给匹配数据包打开 trace，后续 filter/mangle/nat 等路径中的规则命中就有机会被内核记录。

### 2. 在关键终点插 LOG probe

iptables 的 TRACE 日志能显示包经过了哪些 table/chain/rule number，但它不一定能方便地告诉我们“最终到 INPUT 了”还是“POSTROUTING 出去了”。所以项目还会追加三类 LOG 规则：

```sh
iptables -A INPUT   ... -j LOG --log-prefix "NFTP:<runid>:IN "
iptables -A FORWARD ... -j LOG --log-prefix "NFTP:<runid>:FWD "
iptables -t mangle -A POSTROUTING ... -j LOG --log-prefix "NFTP:<runid>:POST "
```

这些 LOG 规则就是 final probe：

- 看到 `NFTP:<runid>:IN`，说明包进入本机。
- 看到 `NFTP:<runid>:FWD`，说明包进入转发路径。
- 看到 `NFTP:<runid>:POST`，说明包到达 POSTROUTING，并能从日志里的 `OUT=` 看到出接口。

### 3. 监听 kernel log

iptables TRACE/LOG 输出都在 kernel log 里。项目会优先使用：

```sh
journalctl -k -f -n 0 -o cat
```

如果没有 journalctl，就退到：

```sh
dmesg -W
```

这里的 `-n 0` 很重要：只跟随后续新日志，不回放历史尾部日志。因为 iptables TRACE 行本身没有 run id，如果回放历史日志，旧的同五元组 TRACE 可能会被误认为本次结果。

### 4. iptables 如何知道命中了哪条规则

iptables TRACE 行通常长这样：

```text
TRACE: filter:INPUT:rule:1 IN=eth0 OUT= SRC=192.0.2.10 DST=198.51.100.20 PROTO=TCP SPT=12345 DPT=443
```

它包含：

- table：`filter`
- chain：`INPUT`
- kind：`rule` 或 `policy`
- rule number：`1`
- 包字段：`IN=`、`OUT=`、`SRC=`、`DST=`、`PROTO=`、`SPT=`、`DPT=`

但是 TRACE 行只给规则序号，不直接给规则内容。所以项目安装完规则后会执行：

```sh
iptables-save
```

然后按每个 table/chain 中 `-A` 规则的顺序建立映射：

```text
table/chain/rule_number -> iptables-save 中的规则文本
```

当 kernel log 里出现 `TRACE: filter:INPUT:rule:1` 时，程序就能反查 `filter/INPUT/1` 对应的规则文本。

相关逻辑在 `internal/trace/parse.go`：

- `parseIPTablesSave` 建立规则号到规则文本的映射。
- `parseIPTablesLogLine` 解析 TRACE/LOG 行。
- `parseIPTablesVerdict` 从规则文本里的 `-j DROP`、`-j ACCEPT`、`-j REJECT` 等字段推断 verdict。

### 5. iptables 如何过滤“属于本次的包”

iptables 的 final LOG probe 有 `NFTP:<runid>:...` 前缀，可以直接判断属于本次运行。

但 iptables TRACE 行本身没有项目写入的 run id，所以项目还会用用户输入的五元组过滤日志：

```text
SRC=<src>
DST=<dst>
PROTO=<proto>
SPT=<sport>  如果用户指定了源端口
DPT=<dport>  如果用户指定了目的端口
```

这也是为什么项目对宽匹配做限制：如果源端口和目的端口都不指定，TRACE/LOG 范围会变大，用户必须显式传 `--allow-broad-match`。

## backend auto 的选择逻辑

`--backend auto` 会探测两个栈：

- `nft --handle list ruleset`
- `iptables --version`
- `iptables-save`

选择策略大致是：

1. 如果 `iptables` 是 legacy，并且 `iptables-save` 里有真实规则，优先用 iptables。
2. 如果 nft ruleset 有真实规则，用 nft。
3. 如果 iptables 有真实规则，用 iptables。
4. 如果两边都没规则，优先选择可用的 nft，再选择 iptables。

这样做是为了避免一些设备上 `nft` 命令存在但实际防火墙规则在 `iptables-legacy` 里，导致选错后端。

## listen 和 active 模式

`listen` 模式只安装探针，然后等待真实流量经过。这是最准确的模式，尤其适合入向包和转发包。

`active` 模式会在安装探针后由本机主动发一个匹配的 TCP/UDP 包：

- TCP：用指定源地址/源端口去 dial 目的地址/目的端口。
- UDP：dial 后写入 `nftracepath\n`。

active 模式只支持 local target，并且不能模拟“从某个入接口进入”的包。如果指定了 `--in-iface`，项目会退化成继续 listen。

## 安全控制

TRACE/LOG 可能产生大量内核日志，所以项目默认加了几层保护：

- 临时规则都带内核侧限速：

```text
nft:      limit rate 10/second burst 20 packets
iptables: -m limit --limit 10/second --limit-burst 20
```

- 用户态最多收集 `--max-events` 条事件，默认 200。
- 超过 `--timeout` 自动退出并清理规则，默认 30 秒。
- 同时省略源端口和目的端口时必须传 `--allow-broad-match`。
- 所有临时规则都有 `nftracepath:<runid>:...` comment 或 `NFTP:<runid>:...` log prefix，方便识别和清理。

## 清理机制

nft 后端清理整张临时表：

```sh
nft delete table inet nftracepath_<runid>
```

iptables 后端按安装时的完整参数逐条 `-D` 删除：

```sh
iptables -D ...
```

清理在 `defer` 中执行，正常结束、超时、错误或 Ctrl-C 都会尝试清理。如果清理失败，结果里会带 `cleanup_error`。

## 当前限制

当前项目有意保持范围较窄：

- 只支持 IPv4 TCP/UDP。
- 端口可以不指定，但范围会变宽。
- NAT 前后五元组关联还没做；结果主要围绕用户输入的原始五元组过滤。
- 不修改 `nf_log`、sysctl、内核模块或系统日志配置。
- 如果目标系统本身不把 TRACE/LOG 输出到可读 kernel log，iptables 后端可能只能超时或给出诊断提示。

## 一句话总结

nft 后端靠 `meta nftrace set 1` 打开内核原生 trace，再用 `nft monitor trace` 的 trace id 和 handle 还原路径；iptables 后端靠 `raw` 表的 `TRACE` target 让内核打印 table/chain/rule number，再用 `iptables-save` 快照反查规则文本，并用额外的 `LOG` probe 判断最终走向。
